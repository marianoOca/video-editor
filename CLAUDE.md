## Pipeline — full flow

```
input/vid*.mp4
  → [1_normalize.py]        → data/combined.mp4  +  data/clip_map.json
  → [2_transcribe.py]       → data/transcript.json
  → [3_analyze.py]          → data/edit_plan.json  (keep = speech blocks; gaps > 0.5s cut, ±0.1s pad; retakes only with --repetitions)
  → [4_render.py]           → data/edited.mp4  +  remotion/src/subtitles.json  +  remotion/public/edited.mp4
                               (then opens Remotion Studio or renders with --render)
  → [4b_place_images.py]    → data/image_plan.json  (skipped if input/images/ is empty)
                               Claude infers timing from transcript, Vision picks placement,
                               injects overlays into Root.tsx defaultProps
  → [5_motion_graphics.py]  → data/graphics_overlay.webm  →  output/final.mp4
                               Ken Burns pan/zoom on images + text lower-thirds via Hyperframes
                               Requires: npx hyperframes-producer (port 9847)
```

Quick command to run everything:
```bash
cd src/pipeline
python3 run_all.py [--from 2] [--model small] [--lang es] [--render]

# Re-run only image placement:
python3 run_all.py --from 5

# Re-run only motion graphics (step 6):
python3 run_all.py --from 6

# Skip lower-thirds or images in step 6:
python3 run_all.py --from 6 --no-lower-thirds
python3 run_all.py --from 6 --no-images
```

## Modes (reel vs youtube)

Step 1 auto-detects orientation from the **first** video in `input/`:

| Input              | Mode      | Target res   | Subtitles |
|--------------------|-----------|--------------|-----------|
| Vertical (W < H)   | `reel`    | 1080 × 1920  | ON        |
| Landscape (W > H)  | `youtube` | 1920 × 1080  | OFF       |

The detected mode is written to `data/mode.json` and read by later steps (no LLM involved — pure file lookup). Override with the `--mode` flag:

```bash
python3 run_all.py --mode reel       # force reel even on landscape input
python3 run_all.py --mode youtube    # force youtube even on vertical input
```

Mixed-orientation inputs are letterboxed (black bars) to the target — no stretching.

## Adding a new video to edit

When user shares a video file: copy it to `input/` (any filename, just `.mp4`), then run pipeline from step 1. Pipeline picks up all `*.mp4` in `input/`. Phone videos with rotation metadata (display matrix -90°) are handled automatically by ffmpeg auto-rotate — no transpose needed.

## Projects (multi-project switching)

Each edit is a **project**: a workspace under `src/data/<name>/` holding all pipeline intermediates for that video. Multiple projects coexist; switching between them is done **inside Remotion Studio** (the sidebar lists one composition per project).

**Project name** = the edited video's filename stem. Precedence: `--project` > stem of `--input` > first-alphabetical stem of `input/*.{mp4,mov}`. Sanitized to a valid Remotion id (`[a-zA-Z0-9-]`).

```bash
python3 run_all.py --input ../../input/prueba3.mov   # → src/data/prueba3/
python3 run_all.py                                    # vid1.mp4,vid2.mp4 → src/data/vid1/
python3 run_all.py --project my-reel                  # explicit name
```

`input/` and `output/` stay **shared** at the repo root (staging). To process a new project, place its video(s) in `input/` (or pass `--input`) and run; afterwards the project lives entirely in `src/data/<name>/` + its Remotion composition, so switching never touches `input/`.

**How switching works:** Steps 4/4b write each project's render-ready snapshot into a multi-tenant Remotion store and regenerate `Root.tsx`:
- `src/remotion/public/projects/<name>/edited.mp4` + `images/` (assets served by `staticFile`)
- `src/remotion/src/projects/<name>.json` (duration/dims/captions/titleCards/imageOverlays)
- `src/remotion/src/Root.tsx` — **generated**: one `<Composition id="<name>">` per snapshot, grouped under a `<Folder name="Projects">`. Open Studio (`npm run dev`) and click a project in the sidebar to switch — instant, no re-run.

**Active project** (which `src/data/<name>/` the pipeline writes to, and the render target) is resolved by: `VE_PROJECT` env var > `.ve_active_project` state file > **first input video alphabetically** (the historical default — the pipeline merges all `input/` videos and names the project after the first). `config.py` hard-errors at import only when none of those exist (input/ is empty too). `run_all.py` sets `VE_PROJECT` from the derived name on a fresh run (`--from 1`), so a new input video overrides a stale state file; on `--from N` (resume) it defers to the state-file project. `config.py` computes `OUT_DIR = src/data/<active>/` at import — this is why the active project must be known before any pipeline import (`run_all.py` sets `VE_PROJECT` before `from config import ...`). Management tools (`project.py`) set `VE_ALLOW_NO_PROJECT=1`, which both tolerates "no active project" and **skips the input-video fallback** so merely listing/switching can't spawn a workspace for a not-yet-a-project video.

**`project.py` CLI** (`src/pipeline/project.py`):
```bash
python3 project.py list             # list projects; active marked *
python3 project.py current          # active project + data dir
python3 project.py switch <name>    # set active (pipeline/render target)
python3 project.py rebuild          # regenerate Root.tsx from snapshots
python3 project.py delete <name>    # remove a project's data + composition
```

**Render** a project: `4_render.py --render` renders the active composition → `output/<name>.mp4` (override stem with `--output-name`). Standalone: `node render.mjs <composition-id>`.

> Architecture note: reviewed `../editor-pro-max` (separate repo). Kept video-editor independent; only borrowed the per-project-composition + Studio-`<Folder>` idiom. We use baked values in a pipeline-generated `Root.tsx` (not `calculateMetadata`), with env-var + state-file active-project persistence.

## Caption architecture

Two independent caption flows:

1. **Pipeline subtitles** (`subtitles.json`): generated by `4_render.py` from `transcript.json`. Mapped to the edited timeline. Read directly in `Composition.tsx` as a static import.

2. **Manually editable captions** (`captions.json` / `captions.srt`): SRT flow for fine-grained editing. Convert with `captions_to_srt.py` → edit the .srt → reconvert with `srt_to_captions.py`.

## Studio "Subtitles" tab (custom Studio chrome via patch-package)

A literal **Subtitles** tab in the Remotion Studio right sidebar (order: Subtitles · Props · Renders),
showing the selected composition's transcript. Remotion has **no public API** for custom Studio tabs
(`registerStudioPanel` = GitHub remotion-dev/remotion#7200, unshipped) — this is a deliberate
`patch-package` hack and **breaks on every Remotion version bump** (re-apply via postinstall). Full
feature roadmap in `HANDOFF-subtitles-tab.md`; deep notes in memory `remotion-studio-custom-tab`.

**Two gotchas that cost real time — do NOT re-discover:**

1. **Patch the CJS file, NOT the ESM chunk.** The Studio frontend bundler forces `@remotion/studio`
   to resolve via its CJS `main`: `getResolveConfig()` in
   `@remotion/bundler/dist/shared-bundler-config.js` has
   `alias['@remotion/studio'] = require.resolve('@remotion/studio')` → `dist/index.js` →
   **`dist/components/OptionsPanel.js`**. Editing `dist/esm/chunk-*.js` has ZERO effect on the frontend.
2. **The bundler treats `node_modules` as managed/immutable.** Its persistent cache
   (`node_modules/.cache/webpack`) ignores content edits to node_modules files, so after the patch
   (re)applies a stale cache serves the OLD bundle. Must `rm -rf node_modules/.cache/webpack` (alone is
   enough). Folded into `postinstall`: `patch-package && rm -rf node_modules/.cache/webpack`.

**How it's wired:**
- Panel component (project-owned, editable without re-patching): `src/remotion/src/studio/SubtitlesTab.tsx`.
- Patched CJS file does `require("../../../../../src/studio/SubtitlesTab")` (bundler resolves the
  cross-boundary `.tsx`; `.tsx` is in `resolve.extensions`). Reads the selected composition's
  `currentDefaultProps.captions`, so it auto-follows composition switches like the Props tab.
- Patch file: `src/remotion/patches/@remotion+studio+<version>.patch` (CJS-only; currently `4.0.478`).
  On a Remotion bump the patch fails to apply (warns, leaves OptionsPanel.js pristine) — re-apply is
  mechanical: re-make the 5 edits on the new `OptionsPanel.js`, `npx patch-package @remotion/studio`,
  delete the old patch, `rm -rf node_modules/.cache/webpack`. Anchors held across 4.0.460→4.0.478. UI copy is **English**
  (matches Studio's native chrome — overrides the "Remotion UI stays Spanish" rule, which applies to
  in-video/editor caption components, not Studio chrome).
- **`captionsEnabled` (display on/off):** the pipeline now ALWAYS builds + stores `captions` in the
  snapshot (every mode, incl. youtube — `4_render.py` builds subtitles unconditionally), so the tab
  is populated everywhere. A separate `captionsEnabled` prop gates ONLY the on-video `<CaptionOverlay>`
  render (`Composition.tsx`): reel→true, youtube→false. The tab has a toggle that flips it via
  `setDefaultProps(..., {shouldSave:true})`. Threaded through `write_project_snapshot(captions_enabled=)`
  + `regenerate_root` (`captionsEnabled: p.captionsEnabled ?? false`). So "youtube = subtitles OFF" in
  the Modes table now means *not burned into the video* — the transcript still exists + shows in the tab.
- **Upgrading Remotion:** run `npx remotion upgrade` **from `src/remotion/`** — the `remotion` bin
  only exists there (via `@remotion/cli`); from the repo root npx can't find it and fails with
  "could not determine executable to run". On the bump, postinstall's `patch-package` fails (the patch
  was made for the old version) — and because postinstall is `patch-package && rm -rf ...`, the `&&`
  short-circuits so the cache clear DOESN'T run. So after re-applying the patch, clear the cache
  manually (`rm -rf node_modules/.cache/webpack`). Stable anchors across bumps: `getSelectedPanel`,
  `onVisualControlsSelected`, the tabs `children:[...]` array, and the body ternary (4.0.478 only added
  `onPointerDown` to the root div).
- Verify without a browser: `curl -s localhost:3000/bundle.js | grep -c "<unique string from the component>"`.

**Sidecar server (`sidecar.py`) — delete captions to cut video:**
The tab's "Apply Cuts" button POSTs to a local Python HTTP server (port 9843) because browser JS
can't shell out to Python directly. The sidecar re-runs `4_render.py --drop-cuts` / `--drop-ranges`
with the deleted caption indices/ranges, producing a new `edited.mp4` in place.

Start it alongside Studio (two terminals):
```bash
# terminal 1
cd src/remotion && npm run dev

# terminal 2
cd src/pipeline && python3 sidecar.py
```

Workflow: scrub video in Studio → Subtitles tab shows transcript in real-time → click ✕ on caption
segments to mark them for deletion → "Apply Cuts" → sidecar re-cuts → Studio hot-reloads the new video.

## Image overlay architecture

Drop images in `input/images/`. Step 4b handles everything:

1. **Claude reads transcript** → decides when each image appears and for how long (anchored to relevant speech segments).
2. **ffmpeg extracts keyframe** at that timestamp from the edited video.
3. **Claude Vision** analyzes face/body position → picks `x`/`y` (0–1 fractions of frame) that avoids covering the subject.
4. Outputs `data/image_plan.json`. Images copied to `remotion/public/images/`.
5. `4_render.py` injects the plan into `Root.tsx` defaultProps as `imageOverlays`.

**In Remotion Studio:**
- Each image renders with 10-frame fade in/out (`interpolate()`).
- Drag the image to reposition — updates `x`/`y` via `updateDefaultProps`.
- Edit `duration_ms` per image in the props panel sidebar.
- Drag handles and dashed borders are Studio-only; not visible in final render.

## Motion graphics architecture (step 6)

Step 6 adds animated overlays on top of `data/edited.mp4` and writes `output/final.mp4`.

**Requires Hyperframes producer running before step 6:**
```bash
npm install -g @hyperframes/cli @hyperframes/producer
npx hyperframes-producer   # stays running in background, port 9847
```

**What step 6 does:**
1. **Ken Burns on images**: reads `data/image_plan.json`, converts each image overlay to a Hyperframes `data-keyframes` + `data-zoom-keyframes` element (pan 3% right, zoom 1.0→1.12 over clip duration).
2. **Text lower-thirds**: Claude reads `data/edit_plan.json` transcript and decides 1–4 moments worth highlighting as animated text overlays (slide-up with fade in/out).
3. Builds `data/composition.html` (standard HTML with Hyperframes `data-*` attributes).
4. POSTs it to `http://localhost:9847/render/stream` → receives transparent WebM (`data/graphics_overlay.webm`).
5. ffmpeg composites overlay onto `edited.mp4` → `output/final.mp4`.

Remotion is **not used** by step 6. Remotion still handles captions + Studio preview independently.

## Caro 15th birthday composition (`src/remotion/src/caro/`)

Standalone Remotion composition (not part of the pipeline). 1920×1080 30fps, ~16 min. Family photo slideshow with intro logo + opening/closing dedication text + photo run with year overlays.

### Layout
- `config.ts` — every timing/dimension knob (single source of truth — change here, not in scenes).
- `theme.ts` — colors + fonts. `colors.logoPink/logoCream/logoGold` sampled from `15caro_text.png`.
- `manifest.ts` — **auto-generated, do not hand-edit**.
- `build_manifest.mjs` — symlinks `input/<year>/` → `public/caro/<year>/`, ffprobes video durations, writes `manifest.ts`. Re-run only when `input/` media changes (changing `PHOTO_DURATION_SEC` does NOT need a re-run — photos use the config constant; videos use baked-in probed durations).
- `components/Background.tsx` — shared `background.png` Img.
- `components/Overlays.tsx` — `CaroLogo`, `YearTag`, `YearTagWindow` (auto-fades at sequence edges).
- `components/FontLoader.tsx` — Great Vibes + Cormorant Garamond via `delayRender`.
- `scenes/IntroLogo.tsx`, `TextScene.tsx`, `MediaItem.tsx`, `PhotoSequence.tsx`.
- `CaroVideo.tsx` — top-level. Series of intro → opening text → photos → closing text. Shared background mounted via standalone `<Sequence>` layers spanning intro+opening and closing.

### Gotchas (learned the hard way — do not re-discover)

1. **Series.Sequence remounts kill cross-scene assets.** Same `<Img src=…>` declared inside two adjacent Series scenes unmounts + remounts at the boundary, flashing 1 frame to the parent background (black) while the image reloads. Fix: hoist persistent assets to the composition root inside their own `<Sequence>` whose `from`/`durationInFrames` span both scenes. This is why `background.png` lives in `CaroVideo.tsx`, not in `IntroLogo` / `TextScene`.
2. **Crossfade requires overlapping `<Sequence>`, not `<Series>`.** `Series` cuts hard between siblings. For overlapping fades, lay out items manually with `from = cumulativeStart - CROSSFADE_FRAMES` and wrap each in an opacity interpolator. See `scenes/PhotoSequence.tsx` (`layoutItems` + `CrossfadeWrapper`).
3. **Overlays repeated per item destroy mount count.** 230+ photos × `<CaroLogo>` + `<YearTag>` = avoidable mounts every 2 s. Group consecutive items by an attribute (year) and render one overlay per range — see `groupByYear` + `OverlayBoundaryFade` in `PhotoSequence.tsx`.
4. **`useCurrentFrame()` inside `<Sequence>` is sequence-local** (`[0, durationFrames]`), not global. Use that for per-item fade-in/out logic; don't try to compute global offsets.
5. **Accumulate-mode "gap between stanzas" ≠ holdEnd → next.** In accumulate text mode the previous stanza stays on screen, so the perceptual pause is `enterEnd → nextStart`. Adding the stanza hold on top makes the gap knob feel broken (hold dominates). The per-stanza hold only matters for the LAST stanza dwell. See `computeStanzaLayout` in `scenes/TextScene.tsx`.
6. **Replace-mode centering breaks if you use `position: absolute` on the stanza inside a flex container.** Render each stanza inside its own `<AbsoluteFill>` so flex-center works per stanza without layout shifts.
7. **Studio HMR shows stale module warnings.** Webpack caches deleted exports after refactors; the warning persists until the next clean build. Ignore if `tsc --noEmit` is clean.
8. **Black-bridge transitions are opt-in per side.** `TextScene` accepts `fadeInFromBlack` / `fadeOutToBlack` (both default `true`). The opening text follows the intro logo on the same `background.png`, so it passes `fadeInFromBlack={false}` — otherwise it fades through black for no reason.
9. **Layer/source order matters for fade-to-black.** The black bridge inside `TextScene` is the LAST child of the outer `<AbsoluteFill>` so it sits above the text. Putting it earlier in the tree means the text is drawn over the black and never disappears.

### Working on this composition

- User scrubs Studio + reports issues as **timestamps** (e.g. "at 00:17.25"). Convert: `frame = round(seconds × 30)`. Map to scene by computing `INTRO_DURATION_FRAMES + BEGINNING_DURATION + …`.
- User often hand-edits `config.ts` between turns to retune values. Re-`Read` it before recomputing anything.
- Never hardcode timing values in scene components. Add a `*_FRAMES` constant to `config.ts` and import it.
- Re-Read `15caro_text.png` and `background.png` if a visual decision depends on them (already in `public/caro/`).
- Studio is started with `npm run dev` from `src/remotion/`. Background log lives in `/tmp/caro_studio.log` during a session.
