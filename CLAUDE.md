## Pipeline — full flow

```
input/vid*.mp4
  → [1_normalize.py]        → data/combined.mp4  +  data/clip_map.json
  → [2_transcribe.py]       → data/transcript.json
  → [3_analyze.py]          → data/edit_plan.json  (silences + retakes detected by Claude)
  → [4_render.py]           → data/edited.mp4  +  remotion/src/subtitles.json  +  remotion/public/edited.mp4
                               (then opens Remotion Studio or renders with --render)
  → [4b_place_images.py]    → data/image_plan.json  (skipped if input/images/ is empty)
                               Claude infers timing from transcript, Vision picks placement,
                               injects overlays into Root.tsx defaultProps
  → [5_motion_graphics.py]  → data/graphics_overlay.webm  →  output/final.mp4
                               Ken Burns pan/zoom on images + text lower-thirds via Hyperframes
                               Requires: npx hyperframes-producer (port 9847)
```

Quick command to run everything:
```bash
cd src/pipeline
python3 run_all.py [--from 2] [--model small] [--lang es] [--render]

# Re-run only image placement:
python3 run_all.py --from 5

# Re-run only motion graphics (step 6):
python3 run_all.py --from 6

# Skip lower-thirds or images in step 6:
python3 run_all.py --from 6 --no-lower-thirds
python3 run_all.py --from 6 --no-images
```

## Adding a new video to edit

When user shares a video file: copy it to `input/` (any filename, just `.mp4`), then run pipeline from step 1. Pipeline picks up all `*.mp4` in `input/`. Phone videos with rotation metadata (display matrix -90°) are handled automatically by ffmpeg auto-rotate — no transpose needed.

## Caption architecture

Two independent caption flows:

1. **Pipeline subtitles** (`subtitles.json`): generated by `4_render.py` from `transcript.json`. Mapped to the edited timeline. Read directly in `Composition.tsx` as a static import.

2. **Manually editable captions** (`captions.json` / `captions.srt`): SRT flow for fine-grained editing. Convert with `captions_to_srt.py` → edit the .srt → reconvert with `srt_to_captions.py`.

## Image overlay architecture

Drop images in `input/images/`. Step 4b handles everything:

1. **Claude reads transcript** → decides when each image appears and for how long (anchored to relevant speech segments).
2. **ffmpeg extracts keyframe** at that timestamp from the edited video.
3. **Claude Vision** analyzes face/body position → picks `x`/`y` (0–1 fractions of frame) that avoids covering the subject.
4. Outputs `data/image_plan.json`. Images copied to `remotion/public/images/`.
5. `4_render.py` injects the plan into `Root.tsx` defaultProps as `imageOverlays`.

**In Remotion Studio:**
- Each image renders with 10-frame fade in/out (`interpolate()`).
- Drag the image to reposition — updates `x`/`y` via `updateDefaultProps`.
- Edit `duration_ms` per image in the props panel sidebar.
- Drag handles and dashed borders are Studio-only; not visible in final render.

## Motion graphics architecture (step 6)

Step 6 adds animated overlays on top of `data/edited.mp4` and writes `output/final.mp4`.

**Requires Hyperframes producer running before step 6:**
```bash
npm install -g @hyperframes/cli @hyperframes/producer
npx hyperframes-producer   # stays running in background, port 9847
```

**What step 6 does:**
1. **Ken Burns on images**: reads `data/image_plan.json`, converts each image overlay to a Hyperframes `data-keyframes` + `data-zoom-keyframes` element (pan 3% right, zoom 1.0→1.12 over clip duration).
2. **Text lower-thirds**: Claude reads `data/edit_plan.json` transcript and decides 1–4 moments worth highlighting as animated text overlays (slide-up with fade in/out).
3. Builds `data/composition.html` (standard HTML with Hyperframes `data-*` attributes).
4. POSTs it to `http://localhost:9847/render/stream` → receives transparent WebM (`data/graphics_overlay.webm`).
5. ffmpeg composites overlay onto `edited.mp4` → `output/final.mp4`.

Remotion is **not used** by step 6. Remotion still handles captions + Studio preview independently.

## Shared modules (pipeline)

- **`config.py`**: `OUT_DIR`, `INPUT_DIR`, `OUTPUT_DIR`, `REMOTION_DIR`, `HYPERFRAMES_PORT`, `probe(path)` (full ffprobe), `get_duration(path)` (duration only). Imported by all steps.
- **`srt_utils.py`**: `ms_to_srt(ms)` and `srt_to_ms(ts)`. Used by `captions_to_srt.py` and `srt_to_captions.py`.

## Language rules

All code and comments must be in **English** — variable names, function names, docstrings, inline comments, print messages, error messages, argparse help strings, documentation.

**Exception:** UI/UX copy in Remotion components (`CaptionOverlay.tsx`, `Composition.tsx`, `schema.ts`) stays in Spanish — it's the editor interface for Spanish-speaking users. Do not translate button labels, error messages shown to users, or zod `.describe()` strings in those files.

## Remotion skill — mandatory for video work

**Any task touching Remotion components, animations, captions, or rendering MUST use the `remotion-best-practices` skill before writing or modifying code.**

The skill is registered and available. Invoke it with:
```
Skill("remotion-best-practices")
```

This applies to: new animations, caption changes, composition edits, schema changes, new Remotion components, render config, and any `remotion/src/` file edits.

For non-video Python pipeline tasks (normalization, transcription, silence detection), the skill is optional.

## Editing preferences & feedback guardrails

User editing preferences are stored in memory at:
`~/.claude/projects/-Users-mar-Documents-Claude-Projects-FA-Automations-video-editor/memory/editing_preferences.md`

**When user gives feedback about an edit** (likes/dislikes a cut, caption style, timing, transition, image placement, pacing, etc.):
1. Recognize it as potential guardrail for future projects.
2. Ask: *"Want me to save this as a guardrail for future projects?"*
3. If yes → append to `editing_preferences.md` using the template (Rule / Why / How to apply / Source + date).

**At start of each project** → read `editing_preferences.md` and apply all active rules before making any edit decisions.
